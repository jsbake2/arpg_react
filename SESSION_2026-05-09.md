## What changed today

### 1. D4 loot filter format — fully decoded

User captured 5 export strings from D4's in-game loot filter
(empty / 1-rule / multi-condition / multi-rule with two recolors /
affix-rich) and saved them to `D4_FILTERS/`. All five decode cleanly
as **base64-encoded protobuf, no compression, no checksums**.

Schema lives in `D4_FILTER_SCHEMA.md`:
- `Filter { rules*, name, slot_id, ? }`
- `Rule { action, color (ARGB fixed32), conditions*, enabled }`
  - action: `0=Show`, `2=Recolor`, `3=Hide`
  - color: ARGB packed; `#FFD700` → `0xFFFFD700`, `#FF3030` → `0xFFFF3030`
- `Condition { type, ids*, amount, ip_max, ? }`
  - type 0 = Item Power (field 5 = upper bound)
  - type 1 = Rarity (field 4 = bitmask: 1=Common, 2=Magic, 8=Legendary)
  - type 4 = Greater Affixes (field 4 = count, field 6 = operator)
  - type 5 = Item Type (field 2 = ID, e.g. `0x6d14c`=Sword, `0x6d16e`=Helmet)
  - types 6 / 7 = Affix groups (one is "required all", other is
    "optional ≥N" — disambiguation deferred until generator built)

`tools/d4_filter_codec.py` — pure-stdlib protobuf encoder/decoder.
All 5 spike samples round-trip byte-identical, which means D4 will
accept anything we encode that conforms to the schema.

**Open work before the build-URL → loot-filter generator ships:**

  [`DiabloTools/d4data`](https://github.com/DiabloTools/d4data) using

### 2. Two launcher fixes for game-daemon mismatch

Symptom: pick POE2 in the launch dialog → panel shows POE2 tabs and
azurite theme, but BUILD tab fetches D4 builds. Two related bugs.

**Fix #1 (commit `4d76cd5`):** the launcher used to spawn the daemon
*before* showing the dialog, so the daemon always came up as D4.
Reordered: QApplication first → dialog → spawn daemon with
`--game <picked>` → panel reuses the same QApplication. Added
`run_panel_with_app(app, ...)` so launcher and standalone-panel
entry points share the panel boot path.

**Fix #2 (commit `7dd35a8`):** even with #1, an existing daemon left
over from a previous launch (or from a manual `arpg-react run`) would
still get attached to with the wrong game. Made it self-healing:
- Daemon writes its game to a marker file at
  `$XDG_RUNTIME_DIR/arpg_react/game` next to the IPC socket.
- Launcher reads the marker before deciding attach-vs-spawn. If the
  running daemon's game ≠ picked game, kill it (SIGTERM, fall back
  to SIGKILL after the standard shutdown grace period) and respawn
  with the right `--game`.

Helpers added in `launcher.py`: `_read_daemon_game(socket_path)`,
`_kill_running_daemon(socket_path)`.

Also (commit `1b4b723`, slightly earlier): the desktop launcher was
hardcoded to `--theme diablo`. Theme follows game now; the desktop
file's Exec line is just `arpg-react app`. `arpg-react install`
defaults to no theme override; `--theme` only baked in when user
explicitly opts in for the NEUTRAL dev palette.

### 4. Hotkey to toggle watcher / auto-cast

Already existed (`pynput.GlobalHotKeys` via
`arpg_react/hotkey.py::HotkeyController`, wired to the
`toggle_watchers` IPC command). Default was `f9`; user changed to
`<alt>+<f9>` via `~/.config/arpg_react/config.json`.

The HotkeyController format parser handles bare keys (`f9`),
pre-formatted (`<f9>`), and combos (`<modifier>+<key>`). Combos must
use angle-brackets around named keys: `<ctrl>+<alt>+<f9>` — not
`ctrl+alt+f9`.

Caveats (already documented in PROJECT.md):
- Pure Wayland without XWayland: hotkey silently disabled, monitoring
  stays always-on, warning in the panel debug console.
- D4 in exclusive fullscreen can swallow `pynput` hotkeys even via
  XWayland. Switch to "Fullscreen Windowed" or "Borderless" if the
  combo doesn't fire in-game.
