### 5. The three deferred items — all closed in one commit (`baab652`)

#### `--game` arg on the daemon
- `arpg-react run --game {d4,poe2}` (default `d4`).
- Plumbed through `cmd_run` → `daemon.run(game=...)` → profile cache
  filename + slot list + detector defaults.

#### Slot enum generalization for POE2
- `HotkeyKind` extended with **Q, E, T, F, LMB, MMB, RMB** as
  distinct enum members. The legacy `LMB = L` / `RMB = R` aliases
  are gone.
- New per-game constants in `config.py`:
  - `HOTKEY_ORDER_BY_GAME` — UI iteration order per game
  - `DEFAULT_KEYMAP_BY_GAME` — installed at daemon startup so D4
    `L → mouse-left` and POE2 `Q → keyboard 'q'` work without a
    user profile cached.
- The `_normalize_button_strings` helper in `rules.py` (which used
  to rewrite `LMB → L` globally) was removed — old D4 builds with
  `"LMB"` strings now deserialize to `HotkeyKind.LMB` and route via
  the D4 default keymap. POE2 builds with `"LMB"` stay distinct.
- **R is shared:** D4 right-mouse and POE2 keyboard 'R' use the
  same enum member; the per-game default keymap disambiguates.

#### Resolution scaling end-to-end
- `scale_detector_for(screen_w, screen_h, ui_scale, base)` in
  `watchers/detector.py` — uniformly scales every pixel coord in
  `DetectorConfig` from the 2560×1440 100% reference. HSV thresholds
  pass through untouched.
- `scale_ocr_bbox(bbox, ...)` + `ocr_bbox_for_profile(game)` in
  `calibrator.py` — same treatment for the POE2 OCR detail-panel
  bbox. Reads cached profile to resolve.
- `scale_template(ref, ...)` in `watchers/detector_refs.py` —
  resizes BOSS_REF / MOUNT_REF pixel patches via Pillow bilinear so
  per-pixel matches still land. Identity at native resolution
  (no allocation).
- Daemon reads `(screen_w, screen_h, ui_scale)` from the cached
  profile, builds a scaled detector + scaled templates, and passes
  them into `Detector(config=, boss_ref=, mount_ref=)`.
- Math assumption documented: uniform scale, 16:9 → 16:9 only.
  Ultrawide (21:9) would need anchor-aware coords; deferred until
  someone shows up on it.

## Next phase — D4 in-game loot filter generator

Asked: "Can we pull items from a build URL and create the in-game
filter for them?"

Answer: **yes if** we can decode D4's loot-filter import-string
format. **No public tool generates these strings today** — d4lf and
similar use their own runtime overlays specifically because the
in-game format is opaque.

A 30-minute spike is on the user's plate. See
`D4_FILTER_DECODE_SPIKE.md` for the exact instructions:

1. User captures **5 progressively richer filter export strings**
   from D4 (empty baseline + 1-rule + multi-condition + multi-rule
   + affix-rich) and pastes them in our session.
2. I run them through `base64 → zlib/gzip → JSON/protobuf`
   detection and report what's inside.
3. From there: either build the
   `Maxroll/Mobalytics URL → affix list → D4 filter rules → encoded
   string` pipeline (~few days), or pivot to plan B (printable
   text recipe — never breaks when Blizzard re-encodes the format).

Either way, the **build-URL → affix-priority parser** is real value
and ships regardless of how the encoding spike ends.
